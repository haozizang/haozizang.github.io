事务隔离

事务

  • 在引擎层实现的(InnoDB)

事务隔离级别

  • 类型

    • 读未提交(read uncommitted)
      • 一个事务还没提交时，它做的变更就能被别的事务看到。
    • 读提交(read committed)
      • 一个事务提交之后，它做的变更才会被其他事务看到。
    • 可重复读(repeatable read)
      • 一个事务执行过程中看到的数据，总是跟这个事务在启动时看到的数据是一致的
    • 串行化(serializable)
      • 是对于同一行记录，“写”会加“写锁”，“读”会加“读锁”。当出现读写锁冲突的时候，后访问的事务必须等前一个事务执行完成，才能继续执行

  • 设置

    • show variables like 'transaction_isolation';

  • 实现

    • 回滚日志 有同一个数据多个不同时刻版本, 不同类型的隔离级别在不同的 read_view
    • 回滚日志在不需要时会被删除

  • Q: 为啥不建议使用长事务?

    • 长事务意味着保留老的事务视图 -> 多回滚记录需要保留; 回滚日志是跟数据字典一起放在 ibdata 文件里的，即使长事务最终提交，回滚段被清理，文件也不会变小。我见过数据只有 20GB，而回滚段有 200GB 的库。最终只好为了清理回滚段，重建整个库。
    • 建议
      • 使用 set autocommit=1 避免造成长事务
      • 显示方式启动事务
      • commit work and chain 来频繁提交+开启下一个事务
    • 定位长事务
      • select * from information_schema.inodb_trx where TIME_TO_SEC(timediff(now(), trx_started) > 60

  • 如何避免长事务的影响?

    • 应用开发端
      • 首先确认是否是 set autocommit=0; 若是则改成 1
      • 不必要的只读事务去掉
      • 基于业务本身的预估, 通过 SET_MAX_EXECUTION_TIME 控制每个语句的最长执行时间
    • 数据库端
      • 监控 information_schema.Innodb_trx 表, 设置长事务阈值, 超过报警/kill
      • percona pt-kill
      • 分析 general_log
