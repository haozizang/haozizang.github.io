索引类型

  • 哈希
    • 查询
      • 缺点: 范围查询需要遍历(因为存储不是有序的)
      • 用于等值查询
    • 插入
      • 冲突处理: 冲突的值组成链表
  • 有序数组
    • 查询较出色
      • 等值查询: 二分法
      • 范围查询: 二分法找到min, 遍历到max
    • 插入
      • 中间插入需要移动, 成本高
      • 因此用于静态存储引擎
  • 搜索树
    • 设计
      • 多叉树-> 降低层数
        • 每个节点对应一个磁盘数据块, 节点跳跃多的话则读磁盘多, 寻址费时
        • eg. innoDB 大约是 1200 叉树, 树高4, 则存 1200^3 个值.树根在内存中, 第二层也很可能在内存中, 访问磁盘次数很少
  • 其他结构
    • 调表
    • LSM树

InnoDB索引模型

  • 索引组织表: 表按照PK顺序以索引形式存放
    • 每一个索引在 InnoDB对应一棵B+树
  • 可以多索引(PK / 非PK)
    • 主键索引(聚簇索引 clustered index): 叶子节点存整行数据
    • 非主键索引(二级索引 secondary index): 叶子节点存PK!
      • 基于非主键索引的搜索需要 再次搜索PK的树

索引维护

  • 插入新行可能场景
    • 直接插入
    • 插入并将其后数据向后挪动, 叶子节点分裂
    • 数据页满-> 页分裂: 申请新的页, 将部分数据移动过去
      • 开销大, 并且影响页的利用率
      • 页合并: 利用率低的页合并的过程

自增字段做索引 VS 自定义的字段做索引

  • 效率
    • PK自增型: 追加操作, 不会移动其他记录, 叶子节点不会分裂
    • 自定义PK: 维持自增有序难, 插入开销大
  • 存储
    • 考虑二级索引的叶子节点为PK的值, PK越小越好!
      • 自增型PK: 4字节(INT) / 8字节(BIGINT)
      • 自定义PK: eg. 身份证字符串 -> 20 个字节
  • 使用业务字段作PK?
    • 唯一索引, 无二级索引(KV场景)

重建索引

  • why
    • 索引被删除, 或页分裂等原因导致的数据页空洞, 重建索引会提高页面利用率, 让索引更紧凑
  • 重建PK? 无意义! 因为 删除/创建主键都会将整个表重建
  • secondary index

回表

  • 从二级索引树找到主键索引, 然后回到主键索引树的过程
  • 避免回表
    • 覆盖索引
      • select * from T where k between 3 and 5; -> select ID from T where k between 3 and 5;
        • 此例中因为 k 的叶子节点存储的就是PK(ID)的值, 不需回表
      • 通过冗余索引(联合索引)达成覆盖 -> eg. 建立 (身份证号, 名字) 的联合索引, 根据身份证号 查找姓名时可避免回表

最左前缀

  • 联合索引的左边部分, 可以加速相应的检索
    • 左边N个字段
    • 字符串左边M个字符
  • 联合索引 (a, b), 则不需要创建 a 的索引
    • 对比 (name, age) + age | (age, name) + name
      • 考虑使用频率(name查的多)
      • 考虑空间(name大)
      • select * from user where name like '张%' and age=10 and ismale=1;
      • 索引下推优化(上面cmd中age会直接在联合索引(name, age)中过滤掉不满足的)
